CJM v0.2.0

Build-time Metadata Compiler for Modern C++

CJM v0.2.0: Practical Models.

Build-time metadata compilation for practical C++ JSON models.

CJM lets developers keep C++ models as the source of truth. It extracts source-level metadata, builds a stable Metadata IR, and generates ordinary C++ integration code at build time.

v0.2.0 expands CJM from a minimal generator into a layered compiler pipeline with practical model support, repeated explicit input headers, and generated-code verification.

Why CJM exists

Modern C++ is powerful. Everyday boilerplate should not be the cost.

Practical schema-shaped models often need repetitive glue for JSON integration, configuration, and metadata-aware workflows.

CJM removes that repetition without replacing C++, adding runtime reflection, or asking teams to adopt a new framework.

Architecture

Compiler architecture, normal compiler workflow.

CJM sits inside the build. The parser/frontend extracts source facts, semantic analysis builds Metadata IR, and the backend consumes that IR to generate ordinary C++. Tree-sitter is being evaluated later as frontend research, not as part of v0.2.

What CJM does

CJM turns ordinary C++ declarations into a stable Metadata IR, then emits ordinary C++ code for the current nlohmann/json backend.

  • Extracts source-level metadata from ordinary C++ declarations
  • Runs richer semantic analysis for practical model mapping
  • Generates nlohmann/json integration code
  • Supports repeated explicit input headers

v0.2.0

The Practical Models release turns CJM into a layered compiler pipeline with practical model support, multi-header CLI input, and stronger verification.

Example

namespace company::model {

struct Address {
    std::string city; // json:"city"
};

struct User {
    std::string name;                    // json:"name"
    int age = 0;                         // json:"age"
    std::vector<std::string> tags;       // json:"tags"
    Address address;                     // json:"address"
    std::optional<std::string> nickname; // json:"nickname,omitempty"
};

} // namespace company::model
cjm generate \
  --input address.hpp \
  --input user.hpp \
  --output model.cjm.hpp
cmake -S . -B build-v02-verify
cmake --build build-v02-verify
ctest --test-dir build-v02-verify \
  --output-on-failure

15/15 tests passed

The current nlohmann/json backend generates ordinary C++ integration code for practical schema-shaped models using Go-style field metadata.

Supported v0.2 model surface

bool signed and unsigned integers floating-point types std::string enums namespaces type aliases nested generated structs std::vector<T> std::optional<T> json:"name" json:"-" json:"name,omitempty"

Known limitations

v0.2.0 intentionally remains a practical subset. CJM does not yet support full C++ grammar parsing or automatic header discovery.

std::map<std::string, T> std::unordered_map<std::string, T> arbitrary dynamic JSON values std::variant / std::any pointer fields custom converters enum string mapping policies time/datetime mappings automatic header discovery full C++ grammar support production parser replacement

Roadmap

v0.3 will focus on practical type coverage, especially string-keyed maps and fixed-width numeric coverage.

v0.3.5 will evaluate Tree-sitter as a future C++ frontend foundation. Tree-sitter is not adopted in v0.2.

Principles

Engineering before magic: CJM should remain early but architecture-driven, useful for practical schema-shaped C++ JSON models, and easy to inspect.

Ordinary C++ should be enough

Users write standard C++. Generated output is standard C++. Everything in between is CJM's responsibility.

Build-time instead of runtime

No hidden runtime, no dynamic reflection engine, and no runtime cost introduced by CJM.

Visible and predictable

Generated C++ should be readable, inspectable, and debuggable when developers need to look inside.

Native to existing builds

Adding CJM should feel like enabling another compiler tool inside an existing CMake project.

Long-term vision

Code generation should feel like the build simply got smarter.

v0.2.0 generates nlohmann/json integration. The larger goal is a reusable compiler foundation for modern build-time metadata workflows in C++: simple user experience, production-quality engineering, and zero runtime overhead introduced by CJM.

CJM succeeds when developers forget that code generation is happening. They write standard C++, run their normal build, and everything works.
